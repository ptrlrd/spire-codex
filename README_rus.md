# Spire Codex

Комплексная база данных и API с игровыми данными **Slay the Spire 2**, созданные путём обратного проектирования файлов игры. Поддерживаются все **15 языков**, поставляемых с игрой.

**Рабочий сайт**: [spire-codex.com](https://spire-codex.com)

**Steam App ID**: 2868840

## Как это было создано

Slay the Spire 2 создана на Godot 4, однако вся игровая логика находится в DLL-библиотеке C#/.NET 8 (`sts2.dll`), а не в GDScript. Конвейер обработки данных:

1. **Извлечение PCK** - [GDRE Tools](https://github.com/bruvzg/gdsdecomp) извлекает файл Godot `.pck`, чтобы получить изображения, анимации Spine и данные локализации (около 9 947 файлов).

2. **Декомпиляция DLL** - [ILSpy](https://github.com/icsharpcode/ILSpy) декомпилирует `sts2.dll` примерно в 3 300 читаемых файлов исходного кода C#, содержащих все игровые модели.

3. **Анализ данных** - 22 Python-парсера на основе регулярных выражений извлекают структурированные данные из декомпилированного исходного кода C# и записывают JSON для каждого языка в `data/{lang}/`:
   - **Карты**: конструкторы `base(cost, CardType, CardRarity, TargetType)` и `DamageVar`, `BlockVar`, `PowerVar<T>` для характеристик
   - **Персонажи**: `StartingHp`, `StartingGold`, `MaxEnergy`, `StartingDeck`, `StartingRelics`
   - **Реликвии/зелья**: редкость, пул и описания, полученные из шаблонов SmartFormat
   - **Монстры**: диапазоны ОЗ, масштабирование по возвышению через `AscensionHelper`, конечные автоматы ходов с намерениями каждого хода (Attack/Defend/Buff/Debuff/Status/Summon/Heal), значения урона, количество множественных ударов (включая шаблоны AscensionHelper), врождённые силы из `AfterAddedToRoom` (42 монстра с вариантами для возвышения), силы, применяемые каждым ходом (цель и величина из `PowerCmd.Apply<T>`), блок, лечение, контекст столкновения (акт, тип комнаты), **схемы атак**, разобранные из `GenerateMoveStateMachine()` (112 монстров - циклические, случайные, условные и смешанные)
   - **Чары**: ограничения по типу карты, возможность суммирования, масштабирование на основе Amount
   - **Столкновения**: составы монстров, тип комнаты (Boss/Elite/Monster), расположение в акте, теги
   - **События**: многостраничные деревья решений (56 из 66 событий), варианты выбора с результатами, расположение в акте, ссылки моделей `StringVar`, преобразованные в отображаемые имена, вычисляемые во время выполнения значения (возрастающие затраты через `GetDecipherCost()`, диапазоны золота через `CalculateVars` с `NextInt`/`NextFloat`, схемы полного исцеления), **предварительные условия** из `IsAllowed()` (25 событий - условия по золоту, ОЗ, акту, колоде, реликвиям и зельям)
   - **Древние**: 8 древних NPC с эпитетами, уникальными для персонажей диалогами, предложениями реликвий и значками портретов
   - **Силы**: PowerType (Buff/Debuff), PowerStackType (Counter/Single), DynamicVars, описания
   - **Эпохи/истории**: данные развития временной шкалы с требованиями для разблокировки
   - **Сферы**: значения пассивного эффекта/активации, описания
   - **Недуги**: возможность суммирования, дополнительный текст карт, описания
   - **Модификаторы**: описания модификаторов забега
   - **Ключевые слова**: определения ключевых слов карт (Exhaust, Ethereal, Innate и т. д.)
   - **Намерения**: описания намерений монстров со значками
   - **Достижения**: условия разблокировки, описания, категории, связь с персонажами и пороговые значения из исходного кода C# (33 достижения)
   - **Акты**: порядок обнаружения боссов, столкновения, события, древние, количество комнат
   - **Уровни возвышения**: 11 уровней (0–10) с описаниями из локализации
   - **Пулы зелий**: уникальные для персонажей пулы, разобранные из классов пулов и ссылок на эпохи
   - **Переводы**: карты фильтров для каждого языка (типы карт, редкости, ключевые слова → локализованные названия) и строки интерфейса (заголовки разделов, описания, имена персонажей) для использования во фронтенде

4. **Формирование описаний** - общий модуль `description_resolver.py` преобразует шаблоны локализации SmartFormat (`{Damage:diff()}`, `{Energy:energyIcons()}`, `{Cards:plural:card|cards}`) в удобочитаемый текст с маркерами форматирования для отображения во фронтенде. Динамические переменные времени выполнения (например, `{Card}`, `{Relic}`) сохраняются как понятные заполнители. Ссылки `StringVar` в событиях (например, `{Enchantment1}` → `ModelDb.Enchantment<Sharp>().Title`) преобразуются в отображаемые имена посредством поиска в локализации.

5. **Рендеринг Spine** - персонажи и монстры представлены скелетными анимациями Spine, а не статическими изображениями. Безголовый рендерер на Node.js собирает позы покоя в портретные PNG размером 512×512. Изображения есть у всех 111 монстров: 100 отрендерены из скелетов Spine, 6 используют общие скелеты по псевдонимам (Flyconid→flying_mushrooms, Ovicopter→egg_layer, Crusher/Rocket→kaiser_crab), а ещё 5 получены из статических игровых ресурсов (Doormaker). Также рендерятся все 5 персонажей (позы в бою, на привале и при выборе персонажа), NPC и фоны. Варианты на основе обликов (Cultists, Bowlbugs, Cubex) рендерятся отдельно. См. раздел [Рендерер Spine](#рендерер-spine) ниже.

6. **Изображения** - портреты карт, значки реликвий и зелий, изображения персонажей, спрайты монстров, значки портретов древних и значки столкновений с боссами извлекаются из игровых ресурсов и раздаются как статические файлы.

7. **Сравнение журналов изменений** - инструмент сравнивает JSON-данные разных версий игры (через ссылки git или каталоги), отслеживая добавленные, удалённые и изменённые сущности каждой категории с различиями на уровне полей. Журналы изменений идентифицируются версией игры в Steam и необязательным номером редакции Codex.

## Общедоступные сервисы

| Хост | Назначение |
|---|---|
| [`spire-codex.com`](https://spire-codex.com) | Общедоступный сайт и API того же источника. Активный бета-канал доступен по пути `/beta`. |
| `cdn.spire-codex.com` | Объектное хранилище Cloudflare R2 для игровых иллюстраций, полных изображений карт, локализованных изображений и архивных бета-ресурсов. |
| [`bot.spire-codex.com`](https://bot.spire-codex.com) | Целевая страница Knowledge Demon и панель сотрудников с аутентификацией через Discord. Бот использует основной API Codex. |
| `analytics.spire-codex.com` | Собственный скрипт и панель Umami. Его база данных PostgreSQL остаётся в частной сети Docker. |
| `tierlists.spire-codex.com` | Выделенное объектное хранилище R2 для созданных изображений предпросмотра списков уровней. |
| `beta.spire-codex.com` | Выведенный из эксплуатации общедоступный хост. Cloudflare перенаправляет запросы на тот же путь корневого домена. |

Хосты CDN и списков уровней являются объектными хранилищами, а не просматриваемыми сайтами, поэтому ответ `404` в корне любого из них ожидаем.

### Локализация

Все игровые данные предоставляются на 15 языках с использованием собственных файлов локализации Slay the Spire 2. Передавайте `?lang=` любой конечной точке данных. Используйте `?channel=beta` для данных активной общедоступной бета-версии; архивные наборы бета-изображений используют `?version=`.

| Код | Язык | Код | Язык |
|------|----------|------|----------|
| `eng` | English | `kor` | 한국어 |
| `deu` | Deutsch | `pol` | Polski |
| `esp` | Español (ES) | `ptb` | Português (BR) |
| `fra` | Français | `rus` | Русский |
| `ita` | Italiano | `spa` | Español (LA) |
| `jpn` | 日本語 | `tha` | ไทย |
| `tur` | Türkçe | `zhs` | 简体中文 |
| `zht` | 繁體中文 | | |

**Что локализовано**: полученные из игры названия и описания сущностей, типы карт, редкости, ключевые слова, силы, столкновения, имена персонажей, заголовки разделов, локализованные маршруты и большинство общих меток интерфейса.

**Что остаётся на английском**: идентификаторы API и структурные значения фильтров, такие как `room_type`, `type`/`stack_type` силы и `pool`, а также названия продуктов и часть редакционного или созданного сообществом содержимого.

Параметры фильтров (`type=Attack`, `rarity=Rare`, `keyword=Exhaust`) всегда используют английские значения независимо от языка - перед сопоставлением бэкенд переводит их в локализованные эквиваленты.

Пример: `GET /api/cards?lang=kor&type=Attack` возвращает данные карт на корейском языке, где тип указан как «공격», корректно отфильтрованные, несмотря на английское значение параметра.

### Форматирование текста

Текстовые поля (`description`, `loss_text`, `flavor`, `text` диалога, `title`/`description` варианта) могут содержать теги в стиле BBCode Godot, сохранённые из данных локализации игры:

| `[green]...[/green]` | Цвет | `[green]healed[/green]` | Текст зелёного цвета |
| `[purple]...[/purple]` | Цвет | `[purple]Sharp[/purple]` | Текст фиолетового цвета |
| `[orange]...[/orange]` | Цвет | `[orange]hulking figure[/orange]` | Текст оранжевого цвета |

| `[jitter]...[/jitter]` | Эффект | `[jitter]CLANG![/jitter]` | Дрожащий анимированный текст |

## Локальный запуск

### Предварительные требования

- Python 3.10+
- Node.js 20+

### Бэкенд

```bash
python -m venv venv
source venv/bin/activate      # Windows: venv\Scripts\activate
pip install -r backend/requirements.txt

cd backend
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Бэкенд работает по адресу **http://localhost:8000**.

### Фронтенд

```bash
cd frontend
npm install
NEXT_PUBLIC_API_URL=http://localhost:8000 npm run dev
```

Фронтенд работает по адресу **http://localhost:3000**.

### Docker

```bash
docker compose up --build
```

Запускает оба сервиса (бэкенд на порту 8000, фронтенд на порту 3000).

### Переменные среды

Основному API только для чтения конфигурация не требуется. Перечисленные ниже необязательные функции

| Переменная | Используется в | Примечания |

## Конвейер обновления

Кроссплатформенный Python-скрипт выполняет весь процесс обновления после выхода новой версии игры:

```bash
# Полный конвейер - извлечение игровых файлов, анализ данных, рендеринг спрайтов, копирование изображений:
python3 tools/update.py

# Указать путь установки игры вручную:
python3 tools/update.py --game-dir "/path/to/Slay the Spire 2"

# Пропустить извлечение (уже имеется свежий каталог extraction/):
python3 tools/update.py --skip-extract

# Только повторно разобрать данные (без извлечения и рендеринга):
python3 tools/update.py --parse-only
```

## Система журналов изменений

Отслеживайте изменения между обновлениями игры с различиями на уровне полей для всех категорий сущностей.

### Создание журнала изменений

```bash
# Сравнить текущие данные со ссылкой git:
python3 tools/diff_data.py HEAD~1 --format json \
    --game-version "0.98.2" --build-id "22238966" \
    --title "March Update"

# Предварительный просмотр в виде текста или Markdown:
python3 tools/diff_data.py HEAD~1 --format text
python3 tools/diff_data.py HEAD~1 --format md
```

### Схема журнала изменений

Каждый JSON-файл журнала изменений содержит:

| Поле | Описание |
|---|---|
| `app_id` | Steam App ID (2868840) |
| `game_version` | Версия игры в Steam (например, `"0.98.2"`) |
| `build_id` | ID сборки Steam |
| `tag` | Уникальный ключ версии (например, `"1.0.3"`) |
| `date` | Дата обновления |
| `title` | Удобочитаемый заголовок |
| `summary` | Количество: `{ added, removed, changed }` |
| `features` / `fixes` / `api_changes` | Подготовленные вручную примечания к выпуску. Они сохраняются при повторном создании существующего тега через `diff_data.py`: различия данных перезаписываются, но эти массивы объединяются с новыми данными. |
| `categories` | Различия по категориям с добавленными, удалёнными и изменёнными сущностями. Изменения полей рекурсивно обрабатываются во вложенных словарях и списках, поэтому каждый конечный элемент получает отдельную строку (например, `vars.DamageVar: 8 → 10`) вместо непрозрачной записи `vars: 2 fields → 2 fields`. |

### Однократная запись и хранение

Файлы в `data/changelogs/` являются историческими записями с однократной записью. `.github/workflows/changelog-guard.yml` блокирует любой PR, который **изменяет или удаляет** существующий журнал изменений. Новые файлы (`A`) разрешены всегда; для изменения PR должен иметь метку `changelog-edit-approved`. Политика и порядок обхода описаны в разделе `CONTRIBUTING.md → Changelog Retention`.

### История отдельных сущностей

`GET /api/history/{entity_type}/{entity_id}` обходит все журналы изменений и возвращает записи, затрагивающие запрошенную сущность, начиная с самых новых. Панель истории версий на каждой странице сведений (`/cards/{id}`, `/monsters/{id}` и т. д.) использует эту конечную точку.

## Развёртывание

### CI/CD (GitHub Actions)

Отправка изменений в `main` запускает `.github/workflows/ci.yml` на собственном исполнителе Kubernetes. Рабочий процесс выполняет поиск секретов, проверки ESLint и TypeScript, линтинг и проверку форматирования ruff, а затем собирает и отправляет стабильные образы с тегом `:latest`. Он также продолжает собирать отдельные бета-образы с тегом `:beta` для `docker-compose.beta.yml`; эти образы сохраняются для эксплуатации, но общедоступные бета-страницы обслуживаются основным развёртыванием по пути `/beta`.

Стабильный фронтенд получает `UMAMI_WEBSITE_ID`. Отдельный бета-образ получает `UMAMI_BETA_WEBSITE_ID`, хотя общедоступный трафик `/beta` использует стабильный фронтенд и его ресурс аналитики.

CI **не** выполняет развёртывание. За него отвечает ежечасное задание автоматического развёртывания на хосте DigitalOcean.

> **Примечание:** `.forgejo/workflows/build.yml` сохранён как неактивный резервный вариант на основе buildah.

### Локальная сборка и отправка

Можно пропустить CI и отправить образы непосредственно со своего компьютера:

```bash
# Собрать и отправить оба образа:
python3 tools/deploy.py

# Только фронтенд:
python3 tools/deploy.py --frontend

# Только бэкенд:
python3 tools/deploy.py --backend

# Проверить сборку без отправки:
python3 tools/deploy.py --no-push

# Пометить выпуск тегом:
python3 tools/deploy.py --tag v0.98.2

# Собрать и отправить бета-образы (тег :beta, без IndexNow):
python3 tools/deploy.py --beta
```

Скрипт автоматически обнаруживает Apple Silicon и выполняет кросс-компиляцию в `linux/amd64` через `docker buildx`. Сначала необходимо выполнить `docker login`.

### Рабочая среда

Общедоступное приложение и сохранённый отдельный бета-стек работают на одном хосте DigitalOcean. Общедоступный трафик использует `spire-codex.com`; на дополнительном хосте Lightsail работает MongoDB.

**Автоматическое развёртывание** - ежечасное задание cron на хосте DigitalOcean запускает `/usr/local/bin/spire-codex-autodeploy` в :03. Когда извлечённый commit изменяется, оно загружает изменения и пересоздаёт как `docker-compose.prod.yml`, так и `docker-compose.beta.yml`, кроме обновлений, ограниченных `data/news/*`. После этого очищается кэш Cloudflare. Журналы записываются в `/var/log/spire-codex-autodeploy.log`. Установка и эксплуатация описаны в [`infrastructure/ansible/README.md`](infrastructure/ansible/README.md).

**Ручное развёртывание**:

```bash
cd infrastructure/ansible
./bin/do-ansible playbooks/deploy.yml

# Сохранённый отдельный бета-стек
./bin/do-ansible playbooks/deploy.yml -e compose_file=docker-compose.beta.yml
```

Рабочие данные подключаются через bind mount (`./data:/data:ro` для фронтенда и в режиме чтения-записи для бэкенда). Новости и состояние забегов читаются из подключённых данных во время запроса, поэтому обновления `data/news/*.json` не требуют перезапуска контейнера.

### Бета-канал (spire-codex.com/beta)

Общедоступное приложение предоставляет стабильные данные и данные Steam `public-beta` как два канала содержимого. Бета-страницы доступны по адресу [`spire-codex.com/beta`](https://spire-codex.com/beta), а локализованные маршруты - по адресу `/{lang}/beta/...`. Основная страница `/images` также предоставляет доступ к архивным версиям бета-ресурсов.

`beta.spire-codex.com` больше не используется публично. В настоящее время Cloudflare выполняет сохраняющее путь перенаправление `302` на корневой домен, но не добавляет `/beta` или `channel=beta`. Поэтому старые ссылки на страницы открывают соответствующую стабильную страницу, а старые запросы API после перенаправления получают стабильные данные. Новые клиенты API должны использовать основной API с явно заданным каналом, например `https://spire-codex.com/api/cards?channel=beta`.

**Архитектура**: `get_channel` преобразует `?channel=beta|stable` в Python `ContextVar`; он также распознаёт заголовок хоста `beta.*` для прямого трафика к источнику. `data_service.py` загружает бета-запросы из `data-beta/<latest>/` и при отсутствии отдельных файлов использует стабильные версии. `GET /api/beta/diff` и `GET /api/beta/version` описывают активную бета-версию, а фронтенд отображает выбранный канал по пути `/beta`.

Отдельный стек `docker-compose.beta.yml` и образы `:beta` по-прежнему собираются и пересоздаются средствами автоматизации развёртывания. Пока активно перенаправление Cloudflare, они не являются общедоступным бета-сайтом.

**Структура данных**: каждая архивная сборка находится в `data-beta/<version>/`, а указатель `latest` выбирает активную сборку. У каждой версии есть собственный каталог `changelogs/`. Архивы бета-изображений повторяют эту структуру в `backend/static/images/beta/<version>/{cards,monsters,misc,ui,vfx}/`.

**Автоматизированный импорт** - `tools/beta-watch/` запускается как задание launchd на Mac разработчика по четвергам с 15:00 до 22:45 каждые 15 минут. Когда SteamCMD сообщает о новом ID сборки `public-beta`, система извлекает и декомпилирует игру, разбирает все языки, создаёт различия, синхронизирует изображения с версиями и открывает PR `auto/beta-<version>`. Установка и эксплуатация описаны в [`tools/beta-watch/README.md`](tools/beta-watch/README.md).

**Ручной импорт**:

```bash
VERSION=vX.Y.Z
PREVIOUS=vA.B.C

# Сначала извлеките и декомпилируйте файлы бета-версии игры, затем выполните анализ из корня репозитория.
(cd backend/app/parsers && \
  EXTRACTION_DIR=../../../extraction/beta \
  DATA_DIR="../../../data-beta/$VERSION" \
  python3 parse_all.py)

VERSION="$VERSION" tools/beta-watch/sync-images.sh

python3 tools/diff_data.py "data-beta/$PREVIOUS/eng" "data-beta/$VERSION/eng" \
  --format json --output-dir "data-beta/$VERSION/changelogs" \
  --game-version "${VERSION#v}" --title "Beta ${VERSION#v}"
```

`sync-images.sh` обновляет символическую ссылку изображений `latest`. PR импорта содержит изменения версионных данных и изображений; после слияния автоматическое развёртывание обновляет оба сохранённых стека.

## Рендерер Spine

Спрайты монстров в StS2 являются скелетными анимациями [Spine](http://esotericsoftware.com/) - каждый монстр представлен файлом `.skel` (бинарный скелет), `.atlas` и таблицей спрайтов `.png`, а не одним изображением. Рендерер собирает их в статические портретные PNG.

### Рендерер WebGL (текущий)

Рендерер WebGL (`render_webgl.mjs`, `render_all_webgl.mjs`) использует **Playwright + spine-webgl** для рендеринга скелетов с помощью GPU безголового Chrome. Благодаря этому получаются чистые изображения **без артефактов треугольных швов**.

**Как это работает:**
1. Запускает безголовый Chrome через Playwright с включённым WebGL
2. Загружает данные скелета, атлас и текстуры в формате base64 на страницу браузера
3. Создаёт холст WebGL, настраивает шейдер spine-webgl и пакетный обработчик полигонов
4. Применяет анимацию покоя и рассчитывает границы (исключая слоты теней и земли)
5. Выполняет рендеринг посредством растеризации треугольников на GPU - без контуров отсечения Canvas и без швов
6. Считывает необработанные пиксели через `gl.readPixels` и переворачивает их по вертикали (WebGL направлен снизу вверх)
7. Записывает PNG через node-canvas, сохраняя прозрачность

**Один скелет:**
```bash
node render_webgl.mjs <skel_dir> <output_path> [size]
node render_webgl.mjs ../../extraction/raw/animations/backgrounds/neow_room ../../backend/static/images/misc/neow.png 2048
```

**Пакетная обработка всех скелетов:**
```bash
node render_all_webgl.mjs  # Рендерит 138 скелетов в backend/static/images/renders/
```

### Покрытие рендерингом

| Категория | Отрендерено | Всего | Примечания |
|---|---|---|---|
| Монстры | 99 | 103 каталога | Изображения есть у всех 111 игровых монстров (99 отрендеренных и псевдонимы/статические изображения) |
| Персонажи | 16 | 16 | Позы в бою, на привале и при выборе |
| Фоны/NPC | 14 | 17 | Neow, Tezcatara, комнаты торговцев, главное меню |
| VFX/интерфейс | 9 | 22 | Для большинства VFX нужны определённые кадры анимации |
| **Всего** | **138** | **158** | Пропущено 20 (нет атласа, только VFX, пустые) |

### Рендерер анимации

Рендерер анимации (`render_gif.mjs`) преобразует анимации покоя и атаки Spine в анимированные WebP, GIF или APNG. Поддерживает варианты обликов, выбор анимации и потоковую запись кадров на диск для крупных анимаций.

**Поддерживаемые форматы вывода:**
- **`.webp`** (рекомендуется) - анимированный WebP без потерь с полной прозрачностью, примерно на 33% меньше APNG. Кадры потоково записываются на диск во избежание нехватки памяти.
- **`.gif`** - 256 цветов, бинарная прозрачность. Самые маленькие файлы, но самое низкое качество.
- **`.apng`** - полная прозрачность, как у WebP, но файлы больше.

```bash
# Отрендерить анимированный WebP без потерь (рекомендуется)
NODE_OPTIONS="--max-old-space-size=8192" node render_gif.mjs <skel_dir> <output.webp> [size] [--fps=N]

# С вариантом облика (для bowlbug, cultists, cubex и т. д.)
node render_gif.mjs <skel_dir> output.webp 256 --fps=10 --skin=rock

# Определённая анимация (по умолчанию: зацикленная анимация покоя)
node render_gif.mjs <skel_dir> output.webp 256 --fps=12 --anim=attack

# Режим белого силуэта (для значков узлов боссов на карте)
node render_gif.mjs <skel_dir> output.webp 256 --white
```

**Библиотека анимаций:** 209 анимированных WebP без потерь:
- 15 анимаций персонажей (бой/выбор/привал × 5 персонажей) размером 512×512
- 103 анимации покоя монстров размером 256×256
- 91 анимация атаки монстров размером 256×256

**Варианты обликов:** 13 монстров имеют варианты обликов (bowlbug, cubex_construct, cultists и т. д.). Для выбора используйте `--skin=`. Облик по умолчанию часто показывает только базовый скелет без тела.

**Шейдер узла босса на карте:** игра использует `boss_map_point.gdshader`, который интерпретирует каналы RGB как маски:
- **Красный канал** × `map_color` (по умолчанию бежевый: `0.671, 0.58, 0.478`) → цвет заливки
- **Синий канал** × `black_layer_color` (по умолчанию чёрный: `0, 0, 0`) → цвет контура
- **Зелёный канал** × белый `1, 1, 1` → блики

### Устаревший рендерер Canvas

Рендерер Canvas (`render.mjs`, `render_all.mjs`) использует `spine-canvas` с `triangleRendering = true`. Из-за сглаживания путей `clip()` Canvas между соседними треугольниками он создаёт **видимые артефакты каркасной сетки**. Вместо него используйте рендерер WebGL.

### Зависимости

- `@esotericsoftware/spine-webgl` ^4.2.107 - среда выполнения Spine для WebGL (текущая)
- `playwright` - безголовый Chrome для рендеринга WebGL
- `gif-encoder-2` - кодирование GIF для рендерера анимации
- `canvas` ^3.1.0 - реализация Canvas для Node.js (буфер кадров рендерера анимации)
- `Pillow` (Python) - собирает WebP/APNG из отрендеренных кадров PNG
- `@esotericsoftware/spine-canvas` ^4.2.106 - среда выполнения Spine для Canvas (устаревшая)

## Версионирование

Spire Codex использует семантическое версионирование **`1.X.Y`**:

| Сегмент | Значение |
|---------|---------|
| **1** | Основная версия Spire Codex (не меняется без полной переработки) |
| **X** | Увеличивается, когда Mega Crit выпускает исправление игры |
| **Y** | Увеличивается при наших исправлениях и улучшениях парсера или фронтенда |

Примеры: `v1.0.0` - первоначальный выпуск, `v1.0.1` - наши исправления ошибок, `v1.1.0` - добавлено первое исправление Mega Crit.

- **Структурированные данные (JSON-LD)**: WebSite + VideoGame (главная), CollectionPage + ItemList (страницы списков), Article + BreadcrumbList + FAQPage (страницы сведений), SoftwareApplication (страница для разработчиков), NewsArticle (news/[gid])
- **Формат заголовка**: `"Slay the Spire 2 (sts2) {Page Title} | Spire Codex"` - стандартизирован для всех страниц. Забеги используют `"{username} - {char} - Ascension {N} {win/loss} - Slay the Spire 2 (sts2) | Spire Codex"`. «(sts2)» находится внутри строки, чтобы соответствовать межъязыковым запросам `sts2 tier list` / `sts2 card list`.
- **Карта сайта**: плоский XML по адресу `/sitemap.xml` с `force-dynamic` (рендерится на сервере, а не во время сборки). Более 20 000 URL, включая страницы сведений о сущностях, страницы матриц просмотра, страницы списков уровней, методику подсчёта очков, сведения runs/[hash] и зеркала i18n для всех типов сущностей
- **Международное SEO**: маршруты `/{lang}/` для 14 языков, кроме английского, с **двунаправленными** альтернативами hreflang - корневые страницы на английском также выводят альтернативы для каждого языка и `x-default` через `buildLanguageAlternates(path)` в `lib/seo.ts` (исправляет кластер дублирующегося содержимого GSC «Crawled - not indexed», в котором Google считал локализованные страницы дубликатами без обратных ссылок)
- **Программное SEO**: 41 страница просмотра карт в `/cards/browse/` (rare-attacks, ironclad-skills и т. д.) и 3 страницы списков уровней (`/tier-list/{cards,relics,potions}`)
- **EntityProse с учётом языка**: страницы сведений отображают короткий абзац на соответствующем языке вместо одинакового английского текста для всех языков
- **Внутренние ссылки**: силы ↔ карты, столкновения → монстры, ключевые слова карт → центральные страницы ключевых слов, ходы монстров → страницы сил (с подсказками), страницы актов → столкновения/события, строки списков уровней → вкладка статистики на странице сущности
- **Open Graph и Twitter Cards**: изображения OG для каждой сущности, карточки Twitter `summary_large_image`
- **Канонические URL**: каждая страница указывает канонический URL

## Технологический стек

- **Бэкенд**: Python, FastAPI, Pydantic, slowapi, сжатие GZip
- **База данных забегов**: MongoDB (статистика сообщества, таблицы лидеров, учётные записи пользователей) с материализованной коллекцией `stats_summary` и фоновым обновлением с выбором лидера. Устаревший путь SQLite сохранён как автономный резервный вариант.

- **Изображения/CDN**: Cloudflare R2 через `cdn.spire-codex.com` (webp)
- **Аналитика и наблюдаемость**: собственный Umami, Prometheus + node-exporter
- **Рендерер Spine**: Node.js, Playwright, @esotericsoftware/spine-webgl (WebGL через безголовый Chrome)
- **Инфраструктура**: Docker, CI GitHub Actions (собственный исполнитель) с кэшем BuildKit в реестре, развёртывание Ansible/SSH
- **Инструменты**: Python (конвейер обновления, сравнение журналов изменений, копирование изображений)
